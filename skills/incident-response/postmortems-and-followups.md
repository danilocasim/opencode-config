# Postmortems and Follow-ups

Postmortems exist to prevent repeats, not to assign blame. Focus on contributing factors, detection gaps, and concrete follow-ups.

## When to load

- The incident is mitigated and you need to write a postmortem.
- You need to define action items and owners.

## When NOT to load

- You are still mitigating user impact.
- You are doing day-to-day refactors.

## Core rules

- Use blameless language.
- Capture a timeline with timestamps.
- Identify contributing factors (not just one “root cause”).
- Include detection/response gaps.
- Action items must have owners and due dates.

## Minimal examples

Sections:

```text
Summary
Impact
Timeline
Contributing factors
What went well / what didn’t
Action items
```

## Anti-patterns

- No action items.
- Action items with no owners.
- Vague “be more careful” conclusions.

## Checklist

- Timeline complete.
- Factors identified.
- Action items owned and dated.
- Learnings communicated.
